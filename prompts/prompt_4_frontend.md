# Frontend Integration: Visual Workflow Editor & AI Prompting

This prompt outlines the next steps for a new agent to implement the frontend user interface for the Factory Workflow Builder. The goal is to create a functional, visually appealing, and interactive visual workflow editor that integrates with the existing Convex backend and AI generation logic.

## Objective
The primary objective is to enable users to:
1.  Submit natural language prompts to generate UI nodes.
2.  Visualize and interact with these nodes on a workflow canvas.
3.  Manage workflows (storing and retrieving configurations from Convex).
4.  Optionally (if time permits/as a bonus), connect nodes with edges.

## Background Context
The following backend components are ready for integration:
*   **Convex Action `generateNode` (`convex/ai.ts`):** Generates a Zod-safe UI node configuration from a natural language prompt.
*   **Convex Mutations & Queries (`convex/nodes.ts`):** Functions to `createNode`, `updateNode`, `deleteNode`, `getNode`, and `getNodesByWorkflow`.
*   **Zod Schema (`convex/schema/nodes.ts`):** Defines the structure of `UINode`, `IOParam`, and `WorkflowGraph`.
*   **Database Schema (`convex/schema.ts`):** Tables for `nodes`, `workflows`, and `edges`.

## Task for the Next Agent

The next agent should implement the following features in the Next.js application:

### 1. Prompt Interface
*   Create a clean and modern prompt input (e.g., a floating search bar or a sidebar panel).
*   Implement the logic to call the `generateNode` Convex action with the user's prompt.
*   Display a loading state while the AI is generating the node.

### 2. Workflow Canvas (Visual Editor)
*   **Editor Container:** A fullscreen or large container acting as the workflow canvas.
*   **Node Rendering:** Dynamically render the generated UI nodes on the canvas. Use the `UINode` structure (label, inputs, outputs, parameters) to determine the layout of each node.
*   **Node Interaction:** Implement drag-and-drop functionality for nodes on the canvas. When a node's position changes, call the `updateNode` mutation to persist the new coordinates in Convex.
*   **Node Styling:** Apply modern styling (e.g., using Vanilla CSS or TailwindCSS as per project convention) to make the nodes look like professional workflow components. Each node should display its `type`, `label`, and its I/O ports.

### 3. State Management & Persistence
*   **Workflow Loading:** On page load, retrieve existing nodes for a specific workflow (or a default one) using the `getNodesByWorkflow` query.
*   **Real-time Updates:** Leverage Convex's real-time capabilities to ensure the UI stays in sync with the database across different sessions/tabs.
*   **New Node Addition:** When a node is successfully generated by the AI, call `createNode` to store it and immediately show it on the canvas.

### 4. Component Refactoring (Optional but Recommended)
*   Create a reusable `Node` component that takes a `UINode` object as a prop and renders its UI.
*   Implement specialized UI components for different node types if `uiComponent` is specified in the node configuration.

## Technical Stack & Constraints
*   **Framework:** Next.js (TypeScript) with React.
*   **Backend:** Convex (use `convex` hooks like `useQuery`, `useMutation`, `useAction`).
*   **Styling:** Prefer Vanilla CSS for node-specific styling or TailwindCSS if it aligns with the existing `globals.css`.
*   **Interactivity:** Use primitive styles at first to validate the output

## Expected Deliverables
*   A fully functional "Prompt-to-Node" workflow.
*   A responsive and interactive visual editor canvas.
*   Persistent node state across page refreshes via Convex.
*   Clear and maintainable React component structure.

This implementation will bring the project from a backend-verified prototype to a user-facing tool for visual workflow construction.
